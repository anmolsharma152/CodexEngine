# Future Memory Model (Research)

> **Status**: Future research. Nothing here is implemented or planned for MVP.
> The current MVP uses `workspace_artifacts` as the sole persistence mechanism.
> This document captures ideas for a future memory architecture — do not implement.

---

## Overview

The current workspace experiment treats artifacts as flat files with a path and
content. A future memory model would add structure: entity types, relationships,
automatic summarization, and agent self-management.

All concepts below are **research notes**, not commitments or requirements.

---

## `memory/` — Agent Self-Managed State

A reserved path prefix in `workspace_artifacts` that the agent uses to store
its own runtime state, similar to how a human keeps notes-to-self.

**Concept:**
- Agent writes observations about the user, the project, and ongoing work to
  `memory/<topic>.md` — without being prompted.
- On session start, agent reads `memory/` artifacts to re-establish context.
- Agent updates memories when facts change (stale detection).

**Open questions:**
- Should memory artifacts have a TTL or be auto-summarized?
- How does the agent distinguish "write for the user" vs "write for myself"?
- Could memory become a feedback loop (agent reinforcing wrong beliefs)?

---

## `skills/` — Reusable Knowledge Packages

Not to be confused with opencode skills. A `skills/` prefix in the workspace
where the agent stores reusable how-to guides and patterns.

**Concept:**
- Agent writes a skill after solving a repeatable problem (e.g., "how to add a
  new search provider").
- Agent reads skills when encountering similar problems in the future.
- Skills are the workspace equivalent of "learned patterns" in memory systems.

**Open questions:**
- How does skill discovery work? Agent reads all skills on boot? Only when
  prompted? Only when a tool call fails?
- How does the agent verify a skill is still accurate?
- What prevents skill bloat (100 skills, most irrelevant)?

---

## `decisions/` — Architecture Decision Records

Structured artifacts capturing why a decision was made, what alternatives were
considered, and what the outcome was.

**Concept:**
- ADRs follow a lightweight template (context, decision, rationale, consequences).
- Agent writes decisions after significant tool-use sequences.
- Decisions are read by the agent to avoid revisiting settled questions.

**Open questions:**
- Should decisions be a separate table or a subtype of `workspace_artifacts`?
- Does the agent need an explicit `log_decision` tool, or is `write_document`
  sufficient with good prompting?
- How does the agent identify "significant" decisions worth recording?

---

## `workspace-state/` — Structured Session Handoff

A machine-readable file (JSON) that captures the agent's current context for
handoff between sessions or subagents.

**Concept:**
- On session end, agent writes a summary of what was accomplished,
  what artifacts were created, and what remains open.
- On session start, agent reads the state file to resume context.
- Could enable cross-session continuity without replaying the full message
  history.

**Open questions:**
- This is the most likely candidate for the MVP's next phase — it solves the
  cold-start problem. But it requires a structured format and a convention for
  when to write/read it.
- Should it be auto-generated by the loop or written by the agent?
- Risk: state file becomes stale or contradicts the agent's actual knowledge.

---

## Repository-Based Memory

Instead of a single file or table, memory is embedded into the workspace like
a git repository of knowledge — with versions, diffs, merges, and branches.

**Concept:**
- Each artifact has a version history (already partially supported via
  `updated_at`).
- Agent can diff two versions of the same artifact to see what changed.
- Multiple agents (or a single agent across sessions) can merge conflicting
  updates to shared artifacts.

**Open questions:**
- Does the workspace need actual versioning, or is "last writer wins"
  sufficient for a single-user app?
- Would git-native versioning (actual commits to a workspace git repo) be
  more robust than a database table?
- How does the agent handle merge conflicts if two sessions modify the same
  artifact?

## Comparison to Existing Systems

| System | Approach | Key Idea |
|---|---|---|
| **MemGPT/Letta** | In-context + external memory | Agent explicitly manages recall and archival |
| **Terminus-2** | 3-step summarization | Proactive context compression at boundaries |
| **LangGraph checkpoint** | Serialized graph state | Full state persistence per step |
| **Ours (current)** | Flat artifacts + query | Agent decides what to write and read |
| **Ours (future)** | Structured memory + auto-summary | Agent manages its own context proactively |

## Guiding Principles (if we build this)

1. **One entity at a time.** Don't add `memory`, `skills`, `decisions`, and
   `notes` simultaneously. Validate each before adding the next.
2. **Measure before optimizing.** The `tool_invocations` table tells us what
   the agent actually reaches for. Build what the data demands.
3. **Path convention over schema.** `analysis/`, `decisions/`, `memory/` are
   just path prefixes in `workspace_artifacts` — no new tables needed until
   relationships or queries prove otherwise.
4. **Agent-driven, not framework-driven.** The agent should decide what
   to remember and recall. The framework just provides the primitives.
5. **No subagents.** Subagents add coordinator complexity. Prefer a single
   agent that can use tools (including its own memory tools).
